项目背景与需求范围

一、项目背景

建设方系建筑装修设计相关企业，需建设/部署 AI 施工实时聊天监控系统，用于提高工地管理效率、压缩沟通与信息差、让现场进度与多方可视可协同。本页为对客版本，不替代双方盖章合同与《方案汇总》全文。

二、核心能力模块（与建设目标对齐）

与内部需求文档一致的五大方向，下文「功能域」与商务分项一一对应。

1. AI 图像识别与比对 设计/施工/效果与现场影像输入；与图纸、上一阶段影像比对，辅助发现偏差、留存各阶段材料。

2. 工作流协同 工人、客户、施工经理、总包等角色之间：任务、问题、整改与验收的闭环，配合消息与待办。

3. 进度与材料（TOC） 材料进出场与使用台账、轻量瓶颈与风险提示（首期为可演示的轻量版，可续深化）。

4. 客户远程观测 以 一客一专属 App + H5/网站/小程序 为主：实时掌握进度/现场/节点，即时工程沟通；日/周、评论、节点与可选摘要，多端同源。

5. 售后 报修、派单、回访的轻量工单与多模态辅助分类（首期为轻量，可二期加强）。

三、技术边界（本方案内可见）

  • 支持 云端或云地混合 部署，具体割接在交付阶段与建设方信息资源衔接。
  • 终端：网页管理后台、微信小程序、Android 安装包、项目 H5/网站；共用业务接口 便于扩展。
  • 识图/多模态：首期为 谷歌多模态模型 + 业务规则；若后续有私有化/换型需求，以补充协议约定。
  • 数据可见范围按 角色与项目/工区 分级，满足多方协同又不放大泄露面。

四、对书面材料的期望（本对客版已覆盖的类型）

  • 建设范围、周期与总价的可读摘要（本页 + 内部《方案汇总》）。
  • 分阶段可见成果与验收方式（下「八周计划」与「商务」）。
  • 运维/增项原则在商务节简述；细项以合同为准。

技术实现方案

1.1 多模态能力

  • 以 谷歌多模态模型 为现网基座，输出结构化结果与业务表对齐，上传走队列与重试/限流策略。
  • 合规模块：分片、水印、按角色脱敏的日志与查询边界。
  • 变化类场景：多期对比 + 规则，关键结论可经人工点确认后再下发。

1.2 系统与终端

  • 管理端、H5/网站、小程序、APK 分端实现，共用后端 API 与项目配置。
  • 服务侧：项目与权限、工作流、文件、多模态代理、WebSocket 聊天与实时通道 等标准模块。
  • 发布：H5/小程序/ APK 可按项目出包与下载/体验入口。

1.3 存储与中间件

关系库、对象存储、缓存、消息队列/异步任务、消息与已读态；满足审计与备份策略落地。

1.4 工作流

典型：派发 → 执行 → 待验 → 整改/复检 → 关闭；异常进入工单与消息网。

1.5 多终端发布与项目专属

管理端维护项目、短链、白标、APK 与小程序的发布/体验方式，保证同项目、多终端、同源业务。

架构主链路

1. 多终端（管理 / H5、小程序、APK、聊天）→ 统一 API 网关与鉴权（RBAC）。

2. 业务服务：项目/任务/材料/售后；文件 进对象存储；即时消息与实时通道 写业务库并推送。

3. 影像类任务经队列调用 谷歌多模态模型，结果过 规则引擎，再进工作流与通知。

4. 进度/TOC/风险以业务库为真源，回推到各端；各端 发版/下载/体验码 与项目配置关联。

接入与数据流（角色侧）

工人、施工经理、客户、总包、售后经同一网关进入业务与文件；异步队列 将影像与多模态任务解耦高峰；规则与异常生成工单，再经工作流与通知多通道触达。全程保留可审计的日志与权限边界，避免「谁都能看全量」的默认配置。
